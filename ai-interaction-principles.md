# AI Interaction Principles: Seqera Run Dashboard

*A short statement of intent for how AI appears and behaves in this interface.
Not a feature list, but a set of design constraints. Each principle maps to a
visible decision in the build, so the behaviour can be inspected, not just claimed.*

## Purpose (who, how, when, why)

- **Who it helps:** a scientist or platform admin triaging a batch of runs,
  specifically at the moment a run **fails** and the raw signal (a truncated
  error banner, an exit code, 50 task rows) does not answer *"what happened and
  what do I do"* fast enough.
- **How:** it compresses the real run data into a plain-language reading that is
  always grounded in fields that exist and always points back to the evidence.
- **When:** on failures and anomalies, on demand. A green run needs no narration.
- **Why:** the artifacts a pipeline emits are built for machines and post-hoc
  forensics, not for a human under time pressure.

## Principles

| # | Principle | Why it matters | How it shows here |
|---|-----------|----------------|-------------------|
| 1 | **Grounded, not magic** | Ungrounded AI erodes trust the first time it's wrong. | Every summary is derived only from fields present in the run (failing process, exit code, retries, duration, cost). It never asserts a cause the data cannot support. |
| 2 | **Show the evidence** | Experts trust what they can verify, not what sounds confident. | The summary cites its source, "see task 38", linking the claim to the row it came from. Reasoning is inspectable in one click. |
| 3 | **Calibrated confidence** | Exit 143 means *killed*, but not *why*. Faking certainty is the classic LLM failure. | When the data supports a diagnosis but not a single cause, it says "most likely," names the top hypotheses (out of memory versus a reclaimed spot instance), and stops there. |
| 4 | **Speaks expert, doesn't dumb down** | The audience is specialists, and condescension is friction. | Concise first, jargon kept ("UNICYCLER", "exit 143"), plain language layered *on top of* the raw signal rather than replacing it. Detail is one level away, not hidden. |
| 5 | **Assistive, human stays in control** | Monitoring is a decision surface, and the scientist owns the call. | AI proposes an interpretation and next steps ("consider raising memory, or rerun from the failed step"). It never auto-acts, never blocks, and never claims to have fixed anything. |
| 6 | **Appears at the pain, quietly** | AI everywhere is noise, whereas AI where it helps is signal. | Surfaces on failed or anomalous runs, not on healthy ones. Expandable and dismissible, an offer rather than an overlay. |
| 7 | **Labelled, always** | Transparency about what is machine-authored is non-negotiable. | Marked "AI summary" wherever it appears, and in this prototype marked **illustrative** to be honest that it is a designed stub, not a live model. |

## Handling thin data (graceful degradation)

The two failed runs in the dataset are different shapes, and the AI must respect that:

- **Died mid-run** (viralrecon: 77 tasks in, assembly step killed) allows a specific,
  evidence-linked diagnosis.
- **Died on arrival** (rnaseq-nf: roughly 8s, zero tasks, no error message) calls for
  only what is honest: *"Failed roughly 8s in with no tasks recorded, most likely a
  launch or config error before execution."* No fabricated detail.

## Prototype versus production

- **In this prototype:** the summary is a static, clearly-labelled stub, generated by
  hand from the real structured fields, to demonstrate the *behaviour* and *tone*
  rather than a live integration.
- **In production:** a Claude API call over the run's structured fields (status, exit
  codes, failing process, retries, resource stats), constrained to the same principles
  above: grounded, cited, hedged, labelled, and versioned.
